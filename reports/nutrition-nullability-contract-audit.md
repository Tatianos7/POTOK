The previous guarded staging apply failed because `public.foods.fiber` is currently `NOT NULL`, while Food Core v02 has no `fiber_100g` values.

- code: `23502`
- message: `null value in column "fiber" of relation "foods" violates not-null constraint`

## App And Runtime Semantics

Current code has mixed fiber semantics:

- `src/services/diaryCreateService.ts` supports nullable fiber in types and snapshot calculation: `fiber: number | null`, with `food.fiber == null ? null : round2(...)`.
- Canonical diary insert preserves nullable snapshot fiber through `DiaryInsertPayload`.
- `src/services/mealService.ts` canonical Supabase read maps DB `fiber == null ? null : Number(...)`.
- Some legacy/local/manual diary paths still normalize missing fiber to `0`, for example `Number(entry.food?.fiber ?? 0)` and `safeNumber(entry.food?.fiber ?? 0)`.
- `src/types/index.ts` and `src/utils/foodNormalizer.ts` model fiber as optional and normalize missing values to 0 in some non-canonical utilities.
- Recipe recompute SQL uses `coalesce(sum(f.fiber * amount_g / 100), 0)`. PostgreSQL `sum` ignores null fiber values, and `coalesce` only turns all-null/no-value totals into 0 for recipe total output.
- `src/services/recipesService.ts` graph reads currently select calories/protein/fat/carbs, not fiber.

Conclusion:

- Nullable `foods.fiber` is compatible with the canonical diary snapshot path.
- There are legacy/local UI paths that may display missing fiber as 0. This is a semantic risk for future fiber-specific UI, but it does not block Food Core import because current primary calories/protein/fat/carbs calculations do not require fiber.
- No code change was made in this audit.

## Final Nutrition Nullability Contract

Canonical required fields:

- `calories` NOT NULL
- `protein` NOT NULL
- `fat` NOT NULL
- `carbs` NOT NULL

Optional field:

- `fiber` NULLABLE with no DB default

Reason:

- `fiber = 0` means confirmed no dietary fiber.
- `fiber = null` means source did not provide a reliable value.
- Food Core v02 does not provide fiber values, so replacing null with 0 would silently convert unknown into false precision.
- Leaving `DEFAULT 0` in the DB would still create false confirmed zero values for future inserts that omit `fiber`, so the default must be removed.

## SQL Patch

Created:

`scripts/sql/staging-make-food-fiber-nullable-draft.sql`

Exact SQL content:

```sql
-- STAGING ONLY
-- DO NOT RUN IN PRODUCTION
-- ADDITIVE/RELAXING DRAFT SQL
-- REVIEW BEFORE EXECUTION
-- TARGET STAGING PROJECT REF: ozidryfvhkcbtpnulakq
-- THIS FILE DOES NOT MODIFY FOOD ROW VALUES
--
-- Nutrition nullability contract:
--   public.foods.fiber is nullable and has no default.
--   NULL means the source did not provide a reliable dietary fiber value.
--   NULL must not be interpreted as confirmed zero.

do $$
declare
  v_fiber_type text;
begin
  if not exists (
    select 1
    from information_schema.tables
    where table_schema = 'public'
      and table_name = 'foods'
  ) then
    raise exception 'public.foods table is required before relaxing fiber nullability';
  end if;

  select data_type
  into v_fiber_type
  from information_schema.columns
  where table_schema = 'public'
    and table_name = 'foods'
    and column_name = 'fiber';

  if v_fiber_type is null then
    raise exception 'public.foods.fiber column is required before relaxing fiber nullability';
  end if;

  if v_fiber_type not in ('numeric', 'real', 'double precision', 'integer', 'bigint', 'smallint') then
    raise exception 'public.foods.fiber must be numeric-compatible, got %', v_fiber_type;
  end if;
end $$;

alter table public.foods
  alter column fiber drop not null,
  alter column fiber drop default;

comment on column public.foods.fiber is
  'Dietary fiber per 100 g. NULL means the source did not provide a reliable value and must not be interpreted as confirmed zero.';

select pg_notify('pgrst', 'reload schema');
```

The file also contains commented validation queries.
